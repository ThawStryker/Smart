# Dirty Worktree 协议

规范路径：`comet/reference/dirty-worktree.md`

此协议由所有可能修改代码的 Comet 子技能共享。当 agent 恢复上下文或继续执行时，必须通过此协议处理未提交的工作树变更。

## 1. 检查

继续或开始代码变更前，运行：

```bash
git status --short
git diff --stat
git diff --cached --stat
git ls-files --others --exclude-standard
```

需要时，检查 `git diff`、`git diff --cached` 和新创建文件的内容。

## 2. 核心规则

- 用户可能不会说明自己改了哪些文件。如果工作树是脏的（包括 Git 状态中显示为 `??` 的新文件），应假定变更可能来自用户或混合来源。
- 脏工作树仅作为代码证据。它不会自动推进 `.comet.yaml` `phase` 或勾选 `tasks.md`；Comet 状态仅在归属确认、验证、所需文档同步以及相关阶段守卫通过后才能推进。

## 3. 归属判断

将脏 diff 分为三类：

1. **属于当前变更**：文件和内容与当前变更目标、tasks.md、plan 或 delta spec 匹配。将 diff 并入当前任务，避免重复工作。
2. **不属于当前变更**：文件或内容不相关。暂停并询问是将其纳入当前变更、拆分为新变更、保持不动，还是在明确授权后丢弃。
3. **来源不明**：diff 和文档不足以判定归属。暂停，报告文件列表和推理，不要推进阶段。

## 4. 常见模式

### 已实现但 tasks.md 未勾选

通过构建和测试验证实现。如果通过，勾选任务。不要因为任务未勾选而重做工作，也不要因为状态文件滞后而忽视代码证据。如果当前子技能定义了阶段特定规则，则遵循该子技能。

### Plan 或 Scope 变更

遵循当前子技能的升级、增量更新或回滚规则。此协议不重复阶段特定细节。

### 恢复意图模糊

当用户说"继续"、"接着来"、"我改了一点"、"不满意"、"重做"、"代码变了"或"用现有的"时，遵循此协议。不要求用户记住具体改了什么。

### open/design 阶段的代码变更

如果当前阶段仍为 `open` 或 `design`，但脏工作树已包含代码变更，首先通过此协议进行归属判断，不要直接推进阶段：

- 如果变更属于当前变更，将其视为需求或设计输入，并相应地记录到 proposal/design/spec/design doc/tasks 中。当前阶段守卫仍必须在进入 build 前通过。
- 如果变更不属于当前变更或归属不明，暂停并询问是纳入、拆分为新变更、保持不动，还是在明确授权后丢弃。
- 不要将 open/design 阶段的代码变更视为已完成实现、可以进入 verify。

## 5. 禁止项

- 在理解脏 diff 来源之前，不得覆盖、还原、格式化或忽略用户变更。
- 不得在脏 diff 未解释的情况下标记验证通过。
